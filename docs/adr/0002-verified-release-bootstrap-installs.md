# ADR 0002 — Bootstrap installs use verified release artifacts

**Status:** Accepted

## Context

Thanos is a Team-grade Governance Layer for Pi. Its README and domain context have treated low-friction bootstrap installation as part of the product shape, but the existing advertised flow pipes `scripts/install.sh` from a mutable branch tip into `sh`.

That creates a supply-chain trust boundary problem: a governance layer should not ask teams to execute unpinned mutable shell content as the normal install path. At the same time, requiring every user to clone the repository before installing would undermine the current distribution model and make the first-run experience heavier than necessary.

The installer also bootstraps Pi when missing and installs or updates the local Thanos harness in `~/.pi`, so failures need to be explicit and auditable rather than silently falling back to weaker behavior.

The main alternatives considered were:

1. **Keep branch-tip `curl|sh`**: preserve the lowest-friction UX but continue trusting mutable branch state.
2. **Require clone-first installation**: avoid remote shell bootstrap, but remove the documented low-friction path.
3. **Use signed release artifacts from day one**: strongest target state, but requires release signing key custody and operational maturity.
4. **Use SHA256-verified GitHub Release artifacts**: preserve bootstrap UX while pinning install bytes to versioned release artifacts and checksums generated by automation.

## Decision

Keep the bootstrap install UX, but install from SHA256-verified GitHub Release artifacts instead of mutable branch tips.

By default, the bootstrap resolves the latest stable release, prints the resolved version, downloads the versioned release source tarball (`thanos-vX.Y.Z.tar.gz`) and the release `SHA256SUMS` asset, verifies the tarball checksum, then installs from the verified payload.

Teams can pin installs and updates with `THANOS_VERSION=vX.Y.Z`. Prereleases are only installable when explicitly requested by version.

`thanos update` updates to the latest stable release by default and also respects `THANOS_VERSION=vX.Y.Z` for pinned updates.

The installer must fail closed when neither `sha256sum` nor `shasum -a 256` is available. It must not fall back to an unverified install.

Release automation generates the release tarball and `SHA256SUMS` after typecheck, lint, and tests pass, before creating the GitHub Release. README/install docs should no longer advertise raw `master` install URLs.

Do not pin Pi from Thanos yet. The installer may auto-install Pi through the package manager already present (`bun` or `npm`), but it should print the installed or detected Pi version. Compatibility warnings can be added later without taking ownership of the user's global Pi runtime.

Do not require signature verification from day one. Document signing as future hardening once release key management is mature.

## Reasoning

- **Preserves adoption path**: Thanos remains easy to install without requiring a local clone first.
- **Removes mutable branch trust**: Install bytes come from a versioned release artifact checked against release-published hashes.
- **Keeps team reproducibility**: `THANOS_VERSION=vX.Y.Z` gives teams a pinning mechanism for automation and policy-controlled environments.
- **Fails safely**: Missing checksum tooling blocks installation instead of silently downgrading the trust model.
- **Avoids premature key ceremony**: Release signing is a meaningful improvement only if key custody, rotation, and publication are handled deliberately.
- **Avoids owning Pi distribution too early**: Printing Pi version gives visibility without fighting user-managed global tooling.

## Consequences

- Release CI must produce and publish deterministic release artifacts and `SHA256SUMS`.
- Installer code becomes a security boundary and needs static validation plus fixture-driven tests for version resolution, checksum verification, and checksum failure behavior.
- Install logs should include release version, artifact URL, checksum URL, computed checksum, install directory, and Pi version.
- Documentation must describe latest-stable bootstrap, explicit `THANOS_VERSION` pinning, checksum verification, and prerelease opt-in behavior.
- Signature verification remains a future ADR or ADR update once signing key management is ready.
